# cat-task-manager

A TUI for managing small, recurring daily tasks. Keyboard-operated and written in Rust.

※ Note: Much of the following text was generated by AI and may be somewhat awkward; I plan to human-edit it later.

This application is not a general-purpose TODO app. It prioritizes the straightforward execution of "fixed, sequential tasks to be done today" over managing projects, deadlines, priorities, tags, search, or complex dependencies.

## Philosophy

Daily tasks often don't stick if their management becomes overly complex.
What's needed is to quickly see the task at hand, record its start time, move to the next task upon completion, and ensure light-touch operations.

Therefore, cat-task-manager does not include features that encourage adding too many tasks. Tasks are not hierarchical, do not have parallel dependencies, nor do they have per-task notes or attributes. To maintain the daily flow, it strongly focuses on progressing through tasks sequentially.

Task definitions and the current day's interim statuses are solely determined by `tasks/*.md`. Tasks are not written in the config. There is no separate status directory. Prioritizing frequent task maintenance (adhering to the ETC principle), the user's interaction point is concentrated into a single text file.

```text
- [ ] 朝のルーティン
- [ ] メール確認
- [ ] コードレビュー
```

This format eliminates the need to duplicate tables or field names when adding tasks. The top-to-bottom order directly dictates the execution sequence, and reordering simply involves moving lines.

## Implementation Philosophy

Task definitions and the current state are consolidated in `tasks/*.md`. A regular non-empty line serves as a task definition, and the state is read as JSON appended to the end of each task line. The end-of-line JSON is an area managed by the application.

```text
- [x] 朝のルーティン {"date":"2026-05-19","state":"done","started_at":"2026-05-19T09:00:00+09:00","completed_at":"2026-05-19T09:05:00+09:00"}
- [ ] メール確認 {"date":"2026-05-19","state":"in_progress","started_at":"2026-05-19T09:12:00+09:00"}
- [ ] コードレビュー {"date":"2026-05-19","state":"not_started"}
```

This is a deliberate design choice to avoid duplicating state into separate files. Even if a typo in a task name is corrected, the state persists as long as the end-of-line JSON for that line remains. If lines are added, deleted, or reordered, the state moves with its corresponding line. Tasks marked `- [x]` without an end-of-line JSON will be normalized by adding an end-of-line JSON with the detected time as both start and completion times. If the end-of-line JSON format is broken, it will be treated as an error upon startup or reload.

Historical data statistics are read from `tasks/*.md` preserved in git history. For the statistics screen, estimated task durations are calculated by removing outliers using the IQR method from past completion records for each task, then displaying the average of the most frequent band from a 5-minute interval histogram. If multiple bands have the same frequency, the median is displayed. Estimated task durations are also shown in the task list. The persistent state for task information is stored solely in `tasks/*.md`.

The display focuses on the next task, while allowing a full overview only when needed. Upon startup, it shows a single line, and `v` toggles between single-line, unfinished, and full views. The unfinished view excludes completed tasks, while the full view includes them. For completed tasks, the working time is shown based on start and completion times. Only tasks that are Not Started, In Progress, On Hold, or Deferred can be interacted with.

The number of states is kept minimal: Not Started, In Progress, Completed, On Hold, Deferred, and Expired. On Hold blocks the next task, while Deferred allows proceeding to the next task without blocking. When a task is "On Hold" in the single-line view of an individual tab, status information is displayed to explain moving to other tabs. When the date changes, unfinished tasks are treated as "Expired" for that day's record. This is a deliberate simplification, not meant to complicate deadlines or scheduling, but simply to record that a task "wasn't finished that day."

Operations are primarily keybinding-centric. Start and complete use the same key, and On Hold (to switch tabs) and resume also toggle with the same key. Deferring tasks is also toggled via key operation. `tasks/*.md` can also be opened via a key command and reloaded upon closing. Rather than building forms within the app, delegating to the user's preferred editor is faster and makes it easier to inspect contents if something goes wrong.

## Configuration Structure

`config.toml` only contains settings for editor candidates, keybindings, and startup git snapshots. No tasks are written here.

```toml
editors = ["fresh", "zed", "nvim", "code"]

[startup_git]
auto_commit_and_push = false

[keybindings]
j = "next"
down = "next"
k = "previous"
up = "previous"
enter = "advance"
space = "advance"
p = "hold"
d = "defer"
q = "quit"
e = "edit"
l = "next_tab"
right = "next_tab"
h = "previous_tab"
left = "previous_tab"
v = "toggle_view"
s = "stats"
"?" = "help"
```

Only when `startup_git.auto_commit_and_push = true`, the application will git commit and push `%LOCALAPPDATA%\cat-task-manager` once a day upon startup.

Save locations are consolidated under Windows' `AppData Local`.

```text
%LOCALAPPDATA%\cat-task-manager\config.toml
%LOCALAPPDATA%\cat-task-manager\tasks\tasks.md
```

`tasks/*.md` is the Single Source of Truth (SSoT) read and written by both the user and the application. `config.toml` defines the operational environment settings.

## Deliberate Simplifications

This application is designed not as a place to "manage tasks," but as a place to "progress through a fixed daily routine."

For needs requiring flexible TODO management, search, tagging, or deadline management, other tools should be used. cat-task-manager begins each day the same way, proceeds sequentially from top to bottom, and records unfinished tasks as the day's outcome. Maintaining this simplicity is an implementation priority.
