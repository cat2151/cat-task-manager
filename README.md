# cat-task-manager

A TUI for managing small, daily recurring tasks. Keyboard-driven. Written in Rust.

※Note: Much of the following text was generated by AI and may be a bit clunky; human review and refinement are planned.

This application is not a general-purpose TODO app. Instead of handling projects, deadlines, priorities, tags, search, or complex dependencies, it prioritizes allowing you to effortlessly proceed with "today's scheduled, ordered tasks."

## Philosophy

Daily tasks tend not to stick if their management becomes overly elaborate.
What's needed is immediate visibility of the current task, a record of its start time, the ability to move to the next task upon completion, and lightweight interaction.

Therefore, cat-task-manager does not include features that encourage over-populating tasks.
Tasks are not hierarchical. They have no parallel dependencies. They also lack per-task notes or attributes.
To maintain the daily flow, it strongly focuses solely on sequential task progression.

Task definitions and their current-day status are solely defined by `tasks/*.md`. Tasks are not written in `config`. It does not maintain a separate status directory.
Prioritizing the ability to frequently maintain tasks, adhering to the ETC principle (Easy To Change), all user-facing task data is consolidated into a single text file.

```text
- [ ] 朝のルーティン
- [ ] メール確認
- [ ] コードレビュー
```

With this format, you don't need to duplicate tables or field names when adding tasks. The top-to-bottom order directly defines the execution sequence, and if you want to change the order, you simply move the lines.

## Implementation Philosophy

Task definitions and their current state are consolidated into `tasks/*.md`. Regular non-empty lines are task definitions, and the state is read as JSON at the end of each task line. The end-of-line JSON is an area managed by the application.

```text
- [x] 朝のルーティン {"date":"2026-05-19","state":"done","started_at":"2026-05-19T09:00:00+09:00","completed_at":"2026-05-19T09:05:00+09:00"}
- [ ] メール確認 {"date":"2026-05-19","state":"in_progress","started_at":"2026-05-19T09:12:00+09:00"}
- [ ] コードレビュー {"date":"2026-05-19","state":"not_started"}
```

This is a deliberate decision to avoid duplicating state into separate files.
Even if you fix a typo in a task name, the state remains intact as long as the end-of-line JSON for that line is preserved.
If lines are added, deleted, or reordered, the state associated with those lines moves along with them.
Tasks marked with `- [x]` but lacking end-of-line JSON will be normalized with the detected time as both the start and completion times in the end-of-line JSON.
If the end-of-line JSON format is malformed, it will be treated as an error during startup or reload.

Only past records are written to `records` by date.
These are not subject to daily operations but represent the results after the day concludes.
To maintain `tasks/*.md` as the Single Source of Truth (SSoT), these records are not used for restoring the current day's state.

The display is designed to focus on the next task while allowing a full overview only when needed.
Upon startup, it defaults to single-line view. Pressing `v` toggles between single-line, incomplete-only, and full views. The incomplete view hides completed tasks, while the full view displays all tasks, including completed ones.
Completed tasks show their elapsed time based on start and completion times. Only tasks that are not_started, in_progress, held, or deferred are interactable.

The number of states is kept small: not_started, in_progress, done, on_hold, deferred, and timed_out.
On_hold blocks the next task, whereas deferred allows proceeding to the next task without blocking.
When an individual tab's single-line view shows a task as 'on_hold', it displays a status message to facilitate moving to other tabs.
When the date changes, uncompleted tasks are treated as 'timed_out' in that day's record.
This is not to complicate deadlines or scheduling, but a pragmatic choice to simply record that "it wasn't finished on that day."

Operations are primarily keybinding-centric. Start and complete use the same key for progression, and hold (to another tab) and resume also toggle with the same key.
Deferring is also toggled on/off via key presses.
`tasks/*.md` can also be opened via a key press, and reloading occurs upon closing the editor.
Delegating task editing to your usual editor is faster than building forms within the app, and it makes reviewing contents easier if something goes wrong.

## Configuration Structure

`config.toml` only holds settings for editor candidates, keybindings, and startup Git snapshot. It does not contain tasks.

```toml
editors = ["fresh", "zed", "nvim", "code"]

[startup_git]
auto_commit_and_push = false

[keybindings]
j = "next"
down = "next"
k = "previous"
up = "previous"
enter = "advance"
space = "advance"
p = "hold"
d = "defer"
q = "quit"
e = "edit"
l = "next_tab"
right = "next_tab"
h = "previous_tab"
left = "previous_tab"
v = "toggle_view"
"?" = "help"
```

Only when `startup_git.auto_commit_and_push = true`, the application will git commit and push `%LOCALAPPDATA%\cat-task-manager` once a day upon startup.

The storage location is consolidated under Windows' `AppData Local`.

```text
%LOCALAPPDATA%\cat-task-manager\config.toml
%LOCALAPPDATA%\cat-task-manager\tasks\tasks.md
%LOCALAPPDATA%\cat-task-manager\records\YYYY-MM-DD.toml
```

`tasks/*.md` serves as the SSoT, read and written by both the user and the application. `config.toml` contains operational environment settings. `records` is treated as the output destination for past results.

## Trade-offs

This application is built not as a "place to manage tasks," but as a "place to progress through a fixed daily routine."

For flexible TODO management, search, tagging, and deadline management, defer to other tools.
cat-task-manager starts the same way every day, progresses sequentially from top to bottom, and records anything left unfinished as that day's result. Maintaining this simplicity is a top implementation priority.
